# solve-skill 证据驱动重构设计

> 日期：2026-06-06
> 主题：将 `solve-skill` 从“读上下文 + 判断状态”重构为“状态路由 + 证据驱动执行控制”
> 状态：设计已确认，待用户审查

---

## 1. 背景与目标

现有 `solve-skill` 已经具备两部分有效能力：

1. 外层任务状态判断；
2. 多步骤任务的流程图契约与节点对照机制。

但它的主执行哲学仍偏向“先读上下文，再根据经验推进”，没有把“查证项目证据”提升为硬门槛。这会让 Agent 在复杂任务中回到旧模式：

```text
读到需求 → 凭经验写代码 → 跑不通再修 → 越修越乱
```

本次重构的目标是把主路径改成：

```text
确认需求 → 查找证据 → 制定计划 → 受控编码 → 复核 diff → 自动验证 → 交付
```

核心目标不是让 Agent “记住项目”，而是让 Agent “查证项目”，并用证据链保证代码正确，而不是用模型记忆保证代码正确。

---

## 2. 重构原则

### 2.1 保留的能力

- 保留外层 8 类任务状态判断，但其职责缩减为路由层。
- 保留多步骤任务流程图机制，并提升为设计阶段的必需产物。
- 保留节点对照块、图外动作停止、阻塞停止、图错误回退确认等规则。
- 保留验证测试阶段的依赖分析思想，但纳入统一验证阶段。
- 保留跨运行时提问兼容机制：优先结构化提问工具，无工具时文本选项兜底并停止等待。

### 2.2 删除或降级的能力

- “读取 `project-context.md` 即可作为主要实现依据”降级为“项目查证时的一个证据来源”。
- “假设优先”不再是实现阶段的默认哲学，只允许用于非阻塞未知和设计阶段临时消歧。
- “先做再问”不再是全局导向，不能再覆盖设计、查证、计划和验证门槛。

### 2.3 总铁律

```text
没有合同，不允许查证。
没有证据，不允许计划。
没有计划，不允许实现。
没有复核，不允许验证。
没有验证，不允许交付。
```

---

## 3. 新的总体结构

新的 `solve-skill` 由两层组成：

### 3.1 外层：任务状态路由

保留以下状态：

1. 想法阶段
2. 需求确认阶段
3. 方案设计阶段
4. 执行实现阶段
5. 验证测试阶段
6. 问题诊断阶段
7. 优化迭代阶段
8. 总结沉淀阶段

这些状态只负责判断“任务当前在哪一类”，不再独立承载实现方法论。

### 3.2 内层：Evidence-Driven 阶段机

只要任务进入以下任一状态，就必须切入统一的证据驱动流水线：

- 方案设计阶段
- 执行实现阶段
- 验证测试阶段
- 总结沉淀阶段

统一流水线为：

```text
Confirm → Locate → Plan → Implement → Review → Validate → Deliver
```

每个阶段都有明确目标、强制产物和禁止继续条件。

---

## 4. Confirm：Requirement Contract

### 4.1 目标

把模糊需求收敛成可执行合同。

### 4.2 必需产物

- 目标
- 包含范围 / 不包含范围
- 参与角色
- 验收标准
- 未确认信息
- 默认假设

### 4.3 停止条件

以下任一情况出现时，不允许进入 `Locate`：

- 目标无法一句话说清；
- 包含范围 / 不包含范围不清；
- 验收标准不可判断；
- 存在阻塞性未确认信息；
- 默认假设会影响整体方案正确性。

非阻塞未知可以继续，但必须记录为默认假设。

---

## 5. 设计阶段必需流程图

### 5.1 定位

对于多步骤任务，流程图不是补充材料，而是设计阶段的必需契约。

### 5.2 要求

进入实现前必须完成并确认：

- Mermaid 业务流程图
- 节点清单
- 初始变更日志

### 5.3 停止条件

以下任一情况出现时，不允许进入 `Implement`：

- 多步骤任务未画流程图；
- 流程图未获得确认；
- 流程图未覆盖完整业务闭环；
- 节点依赖关系不清楚；
- 技术子步骤填不出来。

执行中发现图外动作、节点阻塞、流程图错误时，必须按照流程图契约回退或停止，不能带错执行。

---

## 6. Locate：Evidence Map

### 6.1 目标

查证项目内已经存在的能力、边界与复用点。

### 6.2 必需产物

- 已有入口
- 已有 Service / Repository / QueryService
- 已有 DTO / Type / Entity / Enum
- 类似实现参考
- API / 依赖版本确认
- 证据结论：优先复用项 / 禁止重复创建项

### 6.3 核心规则

- 没有证据，不允许实现。
- 所有新增结构必须先查证项目内是否已有等价实现。
- 证据不足时，允许继续查证，不允许直接编码补洞。

### 6.4 停止条件

以下任一情况出现时，不允许进入 `Plan` 或 `Implement`：

- 没有 Evidence Map；
- 关键依赖、入口、Service、DTO、类似实现未查证；
- 计划新增对象与现有能力重叠，但未说明不复用原因；
- 依赖版本或接口契约存在不确定性，且会影响正确性；
- 证据不足以支撑实现策略。

---

## 7. Plan：Implementation Plan

### 7.1 目标

在编码前固定实现边界、风险与验证方式。

### 7.2 必需产物

- 实现策略
- 计划修改文件
- 计划新增代码
- 计划复用代码
- 不允许修改的文件或目录
- 禁止行为
- 风险分析
- 验证计划

### 7.3 核心规则

- 只允许修改计划内文件。
- 计划外文件如需修改，必须先更新计划。
- 新增代码必须说明原因。
- 未声明验证命令，不允许实现。

### 7.4 停止条件

以下任一情况出现时，不允许进入 `Implement`：

- 没有计划修改文件列表；
- 没有计划复用说明；
- 没有 protected paths；
- 没有 forbidden actions；
- 没有 Validation Plan；
- 已知风险没有处理方式；
- 计划范围超过 Requirement Contract。

---

## 8. Implement：受控编码

### 8.1 目标

在既定计划和流程图契约下进行受控实现。

### 8.2 核心规则

- 只按计划修改；
- 多步骤任务按节点推进；
- 每完成一个节点都必须输出对照块并回写流程图文件；
- 发现图外动作立即停止确认；
- 发现计划外文件需要修改时先更新计划；
- 不允许顺手重构无关代码。

### 8.3 停止条件

以下任一情况必须立刻停止：

- 需要修改计划外文件；
- 需要新增数据库表或字段；
- 需要修改权限模型；
- 需要修改跨服务接口契约；
- 现有 Service 无法满足核心能力；
- 必须绕过模块边界才能完成；
- 出现图外动作；
- 节点被阻塞；
- 同一问题连续修复 3 次仍失败。

---

## 9. Review：Diff Review

### 9.1 目标

检查实际改动是否符合合同、证据和计划。

### 9.2 必需产物

- 实际修改文件
- 实际复用内容
- 实际新增内容
- Diff 检查结果

### 9.3 必查项

- 是否满足需求契约
- 是否复用已有能力
- 是否存在重复 DTO
- 是否存在重复 Wrapper
- 是否违反模块边界
- 是否修改保护文件
- 是否存在无关改动

### 9.4 停止条件

以下任一情况出现时，不允许进入 `Validate` 或 `Deliver`：

- 实际修改文件超出计划；
- 实际新增内容与 Evidence Map / Plan 冲突；
- 出现重复 DTO / Wrapper / Service；
- 出现无关改动；
- 出现越权访问、越模块调用、绕过权限服务；
- 流程图节点完成情况与实际 diff 对不上。

---

## 10. Validate：Verify Result

### 10.1 目标

用命令级验证结果证明代码有效。

### 10.2 必需产物

- 验证项
- 验证命令
- 通过 / 未通过
- 失败说明
- 失败修复记录

### 10.3 核心规则

- 不能只写“已验证”；
- 必须给出命令和结果；
- 验证失败后只能针对失败点修复；
- 修复后必须重新 Review 和 Validate。

### 10.4 停止条件

以下任一情况出现时，不允许进入 `Deliver`：

- 没跑验证命令；
- 没记录验证结果；
- 失败项没有修复记录；
- 类型检查 / lint / 测试未通过；
- 边界检查未通过；
- 重复 DTO / Wrapper 检查未通过；
- 验证失败后做了计划外修复但没更新 Plan / Review。

---

## 11. Deliver：Delivery Report

### 11.1 目标

用明确、诚实、可审计的方式完成最终交付。

### 11.2 必需产物

- 本次完成内容
- 本次未包含内容
- 关键实现说明
- 复用说明
- 验证说明
- 剩余风险
- 后续建议

### 11.3 停止条件

以下任一情况出现时，不能宣布完成：

- 没有完成内容列表；
- 没有未包含内容列表；
- 没有验证说明；
- 没有剩余风险；
- 把默认假设写成已确认事实；
- 把未完成工作包装成已完成或隐性承诺。

---

## 12. 文档结构设计

### 12.1 `skills/solve-skill/SKILL.md`

职责：定义执行规则。

必须包含：

- 新的 skill 目标
- 外层状态路由
- Evidence-Driven 阶段机
- 流程图契约规则
- 提问与确认机制
- 证据规则
- 实现边界规则
- Review / Validate / Deliver 规则
- 总铁律

### 12.2 `docs/skill/evidence-driven-task-template.md`

职责：定义标准任务执行合同模板。

必须承载完整 10 节结构：

- Requirement Contract
- Evidence Map
- Implementation Plan
- Risk Review
- Validation Plan
- Diff Review
- Verify Result
- Delivery Report
- Agent 执行约束

### 12.3 `docs/skill/tasks/YYYY-MM-DD-<task-topic>.md`

职责：复杂任务的运行时实例文档。

要求：

- 复杂任务必须落盘；
- 简单任务可以缩写，但仍需最小 Requirement Contract、Evidence Map、Validate 记录；
- 高风险任务必须完整使用模板。

### 12.4 `docs/skill/flows/YYYY-MM-DD-<topic>-flow.md`

职责：只记录多步骤任务流程图契约。

内容只包含：

- Mermaid 图
- 节点清单
- 变更日志

不得混入完整 Requirement Contract 或 Evidence Map。

---

## 13. 复杂度分层

### 13.1 简单任务

- 可以缩写任务文档；
- 仍必须保留最小 Requirement Contract、Evidence Map、Validate 记录；
- 如果用户明确要求，也可画流程图。

### 13.2 多步骤或跨模块任务

- 必须完整使用任务模板；
- 必须有流程图文件；
- 必须经过完整 Review / Validate / Deliver。

### 13.3 高风险任务

- 完整任务模板；
- 完整流程图；
- 明确 protected paths；
- 全量验证记录；
- 任一架构边界变化必须停止等待人工确认。

---

## 14. 迁移策略

### 14.1 本次重构包含范围

- 重写 `skills/solve-skill/SKILL.md`
- 新增 `docs/skill/evidence-driven-task-template.md`
- 在 `SKILL.md` 中定义 `docs/skill/tasks/` 与 `docs/skill/flows/` 的职责
- 保留并整合现有流程图机制

### 14.2 本次重构不包含范围

- 批量改造其他 skill
- 为历史任务补写 task 文档
- 编写任务合同自动生成脚本
- 引入外部依赖、浏览器渲染或 Mermaid CLI

### 14.3 迁移顺序

1. 重写 `SKILL.md` 主结构。
2. 新增证据驱动任务模板文档。
3. 定义运行时任务文档路径与规则。
4. 明确任务文档与流程图文档的职责边界。
5. 删除旧的冲突表述，例如：
   - 以“读上下文”替代“查证项目”；
   - 以“先做再问”覆盖设计和实现门槛；
   - 把某个特定提问工具名写成唯一合法机制。

---

## 15. 成功标准

- `solve-skill` 不再要求 Agent 记住项目，而要求 Agent 查证项目。
- 多步骤任务在设计阶段必须先有并确认流程图。
- 进入实现前必须存在 Requirement Contract、Evidence Map、Implementation Plan。
- 实现阶段只能按计划内文件受控编码。
- 完成阶段必须具备 Diff Review、Verify Result、Delivery Report。
- 缺少证据链任一关键产物时，不能宣称完成。
- 新规则不依赖单一运行时工具名，能兼容 Claude 与 Codex。

---

## 16. 设计结论

本次重构完成后，`solve-skill` 将从“状态判断 + 建议式执行”升级为“状态路由 + 证据驱动执行控制”。

它的核心不再是“告诉 Agent 可以怎么做”，而是“规定 Agent 在什么证据条件下才允许继续”，并把流程图契约、项目查证、受控编码、diff 复核、自动验证和最终交付统一纳入一条可审计的执行链。
