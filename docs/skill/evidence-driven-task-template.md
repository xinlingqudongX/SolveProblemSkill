# 证据驱动任务执行模板

> 适用于复杂任务、多步骤任务和高风险任务。
>
> 使用方式：把本模板复制为具体任务文档后逐段填写。每个主段都需要留下可审计内容；若某项不适用，必须写明“不适用”和判断原因，不能留空。

## 0. 多步骤任务附加要求

填写说明：本段仅用于多步骤任务，作为进入 Implement 前的额外契约。若任务包含多个有依赖关系的步骤、节点或阶段，必须先补齐本段，再进入后续执行。

- 多步骤任务在 Implement 前必须有已确认的流程图文件。
- 流程图文件必须包含 Mermaid 图、节点清单、变更日志。
- 执行中必须按节点对照块推进，并在每次节点状态变化后回写流程图文件。
- 出现图外动作、节点阻塞、流程图错误时必须立即停止并处理；未处理前不得继续执行后续节点。

至少记录：流程图文件路径、当前确认版本、节点负责人或执行者，以及最后一次回写时间。若任务不是多步骤任务，也要明确写“本任务不适用流程图文件契约”。

## 1. Requirement Contract

填写说明：本段用于固定任务目标、边界和验收口径，避免执行中目标漂移。最小完成要求是让任何协作者在不追问上下文的情况下，能判断“要做什么、不做什么、做到什么算完成”。

### 1.1 目标
- 任务目标：
- 预期业务结果：

至少记录：一句话任务目标和一句话结果定义。目标必须可判断是否达成，不能只写“优化一下”“处理一下”。

### 1.2 需求范围
- 包含范围：
- 不包含范围：
- 边界条件：

填写说明：明确本次改动覆盖的对象、模块和行为，同时写清不在本次处理范围内的内容。最小完成要求是至少记录 1 条包含范围和 1 条不包含范围。

### 1.3 参与角色
- 用户：
- Agent：
- 相关系统 / 服务：

至少记录：谁提出需求、谁执行、会影响哪些系统或服务。若没有外部角色，也要写明仅本地 Agent 执行。

### 1.4 验收标准
- 功能验收：
- 约束验收：
- 交付验收：

通过条件：每条验收标准都必须能被验证命令、页面行为或 diff 结果直接判断。不能使用“看起来正常”这类不可审计表述。

### 1.5 未确认信息
- 阻塞性未确认信息：
- 非阻塞性未确认信息：

必须记录：阻塞性未确认信息是否存在，以及存在时为什么会阻塞执行。阻塞性确认不能跳过；若为空，也要写“无阻塞性未确认信息”。

### 1.6 默认假设
- 假设 1：
- 假设 2：

填写说明：这里只能记录为了推进任务而采用的临时假设，不能把未确认事实写成已确认信息。至少记录会影响实现或验证方式的关键假设。

## 2. Evidence Map

填写说明：本段用于证明实现依据来自项目证据，而不是模型记忆。最小完成要求是覆盖入口、复用点、类型结构、类似实现和依赖约束中的实际证据来源。

### 2.1 已有入口
- 入口文件 / 模块：
- 调用链起点：

至少记录：实际查到的入口文件、模块或命令入口。若没有直接入口，也必须记录查找范围和未找到的结论。

### 2.2 已有 Service / Repository / QueryService
- 可复用 Service：
- 可复用 Repository：
- 可复用 QueryService：

填写说明：优先写可直接复用的对象，其次写相邻能力和缺口。最小完成要求是说明“找到什么”“准备怎么复用”或“为什么不能复用”。

### 2.3 已有 DTO / Type / Entity / Enum
- DTO：
- Type：
- Entity：
- Enum：

必须记录：与本次改动直接相关的数据结构证据。即使某类对象不存在，也要写明“未发现现成结构，需要新增”的判断依据。

### 2.4 类似实现参考
- 相似功能：
- 可对齐实现：

填写说明：引用项目内最接近的实现方式，作为行为、命名或测试写法的参考。至少记录 1 个相似实现或明确写出未找到类似实现。

### 2.5 API / 依赖版本确认
- 外部 API：
- 内部接口契约：
- 依赖与版本：

最小完成要求：确认会影响实现的接口契约或依赖版本。若本任务不涉及外部 API，也必须记录“本任务不涉及外部 API”。

### 2.6 证据结论
- 复用结论：
- 新增结论：
- 约束结论：

填写说明：把前面分散的证据收束成执行结论。至少记录哪些能力直接复用、哪些必须新增、哪些约束会影响实现路径。

## 3. Implementation Plan

填写说明：本段用于把“准备怎么做”具体化到文件、步骤和限制条件。最小完成要求是任何人按本段都能知道应修改哪些文件、避免哪些动作、先做哪一步后做哪一步。

### 3.1 实现策略
- 核心方案：
- 兼容策略：

至少记录：核心实现路径和兼容/回退考虑。不能只写“按常规实现”，必须说明采用该方案的理由。

### 3.2 计划修改文件
- 修改文件：
- 新增文件：

必须记录：计划内修改文件清单。若执行中发现计划外文件，需要先更新本段，再继续实现。

### 3.3 计划复用代码
- 直接复用：
- 参考复用：

填写说明：把复用对象与复用方式写清楚，避免实现阶段重新发明已有能力。至少记录 1 条复用结论或明确写出本次无法复用的原因。

### 3.4 Forbidden Actions / Protected Paths
- Forbidden Actions：
- Protected Paths：

最小完成要求：列出本次明确不能做的事情，以及不能被误改的路径。若没有特殊 protected path，也应记录默认只允许改计划内文件。

### 3.5 执行顺序
- Step 1：
- Step 2：
- Step 3：

填写说明：执行顺序必须反映依赖关系，而不是随意罗列步骤。至少记录从查证结果到改动落地再到验证准备的顺序。

## 4. Risk Review

填写说明：本段用于提前暴露失败点和回退面，避免把风险留到验证后才发现。最小完成要求是写出最可能影响交付的风险和对应缓解动作。

### 4.1 功能风险
- 风险点：
- 缓解措施：

至少记录：功能行为可能出错的位置，以及如何发现或降低该风险。

### 4.2 兼容性风险
- 影响范围：
- 回退策略：

填写说明：如果改动会影响现有调用方、配置或数据格式，必须在这里写清范围和回退办法。

### 4.3 依赖风险
- 外部依赖：
- 版本约束：

最小完成要求：记录会影响执行结果的依赖项、工具链或版本约束。没有依赖风险时写“未发现额外依赖风险”。

## 5. Validation Plan

填写说明：本段在编码前定义“怎么证明有效”。最小完成要求是给出验证项、验证命令和可判断的通过条件，不能只写“跑一下测试”。

### 5.1 验证项
- 功能验证：
- 回归验证：

至少记录：本次改动的直接验证项，以及最容易被影响的回归点。若任务很小，也不能省略最小验证。

### 5.2 验证命令
- 命令 1：
- 命令 2：

必须记录：实际准备运行的命令、检查动作或人工验证步骤。命令应尽量具体到文件、测试目标或页面行为。

### 5.3 通过标准
- 通过判定：
- 失败处理：

通过条件：必须能根据命令输出、页面结果或文件状态直接判断通过。失败时记录需要说明失败点、影响范围和下一次修复目标。

## 6. Diff Review

填写说明：本段用于在改动完成后核对实际结果是否仍然符合合同和计划。最小完成要求是把“改了什么、复用了什么、有没有偏离计划”记录清楚。

### 6.1 实际修改文件
- 文件清单：

必须记录：实际修改或新增的文件，不能只写“见 diff”。若实际改动超出计划，需要明确标注偏差原因。

### 6.2 实际复用内容
- 复用项：

至少记录：最终真正复用到的模块、函数、结构或模式。若计划复用但最终没有复用，也要说明原因。

### 6.3 Diff 检查结果
- 合同一致性：
- 计划一致性：
- 无关改动检查：

填写说明：逐项判断是否满足 Requirement Contract、是否落在计划范围内、是否引入无关改动。最小完成要求是给出明确结论，而不是只写“已检查”。

## 7. Verify Result

填写说明：本段用于记录实际验证执行结果，而不是验证计划。最小完成要求是写出跑了什么、结果如何、失败后怎么修，再给出最终通过与否。

### 7.1 验证记录
- 验证命令：
- 结果：

必须记录：实际执行的命令或检查动作，以及对应结果摘要。结果不能只写“通过”，应说明通过的是哪一项。

### 7.2 失败修复记录
- 失败点：
- 修复动作：

失败时记录：第一次失败出现在哪里、采取了什么修复、修复后重新验证了什么。若全过程无失败，也要写“本轮验证未出现失败修复项”。

### 7.3 最终结论
- 是否通过：
- 剩余问题：

通过条件：只有当验证记录支撑所有必需验证项时，才能写“通过”。若存在未覆盖项或残留问题，必须明确写入剩余问题。

## 8. Delivery Report

填写说明：本段用于把最终交付结果整理成用户可消费、可追溯的结论。最小完成要求是清楚区分已完成、未包含、验证情况和剩余风险。

### 8.1 本次完成内容
- 已完成：

至少记录：本轮实际交付的功能、文档或约束变更，避免把计划中的事项误写成已完成。

### 8.2 本次未包含内容
- 未包含：

必须记录：明确不在本次交付中的内容，特别是用户可能误以为已覆盖的部分。

### 8.3 关键实现说明
- 实现摘要：

填写说明：用简洁语言概括关键实现路径、核心复用点或重要取舍，不要求重复完整实现细节。

### 8.4 验证说明
- 已执行验证：

至少记录：最终交付前实际完成了哪些验证，以及这些验证覆盖了什么范围。

### 8.5 剩余风险与后续建议
- 剩余风险：
- 后续建议：

必须记录：当前仍然存在但可接受的风险，以及下一步更完整交付时的建议动作。不能把未完成工作伪装成已完成。

## 9. Agent 执行约束

填写说明：本段用于提醒执行者哪些门槛绝不能跳过。最小完成要求是把进入下一阶段的条件、记录要求和升级条件保留下来，作为执行时的硬约束。

### 9.1 执行门槛
- 没有 Requirement Contract，不进入 Locate。
- 没有 Evidence Map，不进入 Plan / Implement。
- 没有 Validation Plan，不能完成 Plan / 不能进入 Implement。
- 没有 Review，不进入 Validate。
- 没有 Validate，不进入 Deliver。

本模板阶段映射约定：Review 指 Risk Review 与 Diff Review 的必填审查内容；Validate 指 Verify Result；Deliver 指 Delivery Report。多步骤任务还必须满足“流程图文件已确认且可执行”这一前置条件，否则不得进入 Implement。

### 9.2 记录要求
- 每一步必须留下可审计记录。
- 验证必须记录命令与结果。
- 假设必须与已确认事实分开描述。

- 多步骤任务执行时必须按节点对照块逐项推进，并同步更新流程图文件中的 Mermaid 图、节点清单、变更日志。
- 若出现图外动作、节点阻塞、流程图错误，必须先记录并处理，再决定是否更新计划、审查或验证路径。

必须记录：命令、结论、偏差和阻塞处理。任何“默认知道”的背景信息都不等于记录。

### 9.3 升级条件
- 发现阻塞性未确认信息时暂停推进。
- 发现计划外高风险改动时先更新合同与计划。
- 验证连续失败时回到 Evidence Map 或 Risk Review 重新审视。

失败时记录：触发升级的原因、暂停点和下一步需要补齐的证据或确认项。
